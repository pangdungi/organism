Kpi.js 기준 · 백엔드에 사용자 아이디별로 KPI 데이터가 저장되는 방식

1. Kpi.js에서 사용하는 데이터 구조 (프론트엔드)

collectTasksFromDOM()이 반환하는 task 객체

필드명 한글명 타입 설명
categoryId 카테고리ID string dream | sideincome | happiness | health
status 상태 string 진행중 | 시작안함 | 하기로결정함 | 달성완료 | 달성실패
name 목표명 string 목표 설명
startDate 시작일 string YYYY-MM-DD
dueDate 마감일 string YYYY-MM-DD
kpi KPI 설명 string KPI 설명
actual 실제값 string 현재 달성값
target 목표값 string 목표값
memo 메모 string 사이드 패널 메모

2. Kpi.js 카테고리/상태 값

구분 id / value label (표시)
카테고리 (CATEGORIES) dream 꿈(원하는 삶)

health 건강

하기로결정함 하기로 결정함
달성완료 달성완료
달성실패 달성실패

3. 백엔드 컬럼 정의 (life_kpis 테이블)

Kpi.js 필드 → DB 컬럼 매핑

컬럼명 한글명 타입 필수 Kpi.js 매핑
id ID UUID O 고유키 (자동생성)
user_id 사용자ID UUID O 로그인 사용자 (auth.users.id)
category 카테고리 string O ← task.categoryId (dream/sideincome/happiness/health)
goal_name 목표이름 텍스트 O ← task.name
start_date 시작일 날짜 X ← task.startDate
due_date 마감일 날짜 X ← task.dueDate
status 현재상태 string O ← task.status (진행중/시작안함/하기로결정함/달성완료/달성실패)
kpi_name KPI이름 텍스트 X ← task.kpi
actual_value 실제값 텍스트 X ← task.actual
target_value 목표값 텍스트 X ← task.target
memo 메모 텍스트 X ← task.memo (사이드 패널 메모)
parent_goal_id 상위목표ID UUID X 하위 목표일 때 상위 id (+ 하위 버튼)
created_at 등록일시 날짜시간 O 자동 생성
updated_at 수정일시 날짜시간 O 자동 수정

user_id category goal_name start_date due_date status kpi_name actual_value target_value parent_goal_id
user_abc123 dream 누구나 따라할 수 있는 "시스템형 시간관리법" 프레임워크를 완성한다 2026-01-01 2026-02-15 진행중 프레임워크 진척률(%): 0→100 10% 100% —
user_abc123 dream 타겟 독자 정의(페르소나 2-3개)+문제/해결 메시지 한 문장으로 — — 시작안함 최종 파일 품질 체크리스트 통과율: 목표 95%+ 0 95% 위 행의 id
user_abc123 dream 타겟 3개 포트폴리오 만들기 — — 시작안함 (세부계획 참조) 0 3 위 행의 id
user_abc123 sideincome 2026년 안에 대출 상환 완료한다 2026-02-28 2026-03-17 하기로결정함 대출잔액 0원 0원 0원 —
user_abc123 sideincome 신용카드를 사용하지 않는다 2026-02-18 — 진행중 신용카드 사용 안 한 날 30일 1 30일 —
→ 같은 user_id로 여러 행이 쌓이고, category(dream/sideincome/happiness/health)와 parent_goal_id로 상위·하위 목표를 구분합니다. (Kpi.js CATEGORIES, STATUS_OPTIONS 기준)

시간가계부 테이블 구조

Time.js 기준 · 백엔드에 사용자 아이디별로 시간가계부 데이터가 저장되는 방식

1. Time.js에서 사용하는 데이터 구조 (프론트엔드)

collectRowsFromDOM()이 반환하는 row 객체

필드명 한글명 타입 설명
taskName 과제명 string 과제/업무 이름
timeTracked 사용시간 string hh:mm 형식 (예: 01:30)
category 카테고리 string dream | sideincome | happiness | health | pleasure | dreamblocking | unhappiness | unhealthy | moneylosing | work | sleep
productivity 생산성 string productive | nonproductive | other (카테고리 선택 시 자동 매핑)
date 날짜 string YYYY-MM-DD
feedback 피드백 string 자유 메모

※ 가격(price)은 시급 × 사용시간 × 생산성 계수로 클라이언트에서 계산. 저장 시 계산된 값을 함께 저장하거나, 시급을 사용자별 설정으로 두고 조회 시 계산 가능.

2. Time.js 카테고리/생산성 값

구분 value label (표시)
카테고리 (CATEGORY_OPTIONS) dream 꿈
sideincome 부수입
happiness 행복
health 건강
pleasure 쾌락충족
dreamblocking 꿈을 방해하는 일
unhappiness 불행
unhealthy 비건강
moneylosing 돈을 잃는 일
work 근무
sleep 수면
생산성 (PRODUCTIVITY_OPTIONS) productive 생산적
nonproductive 비생산적
other 그외

3. 카테고리 → 생산성 자동 매핑

생산적 (productive) 비생산적 (nonproductive) 그외 (other)
dream, sideincome, happiness, health unhappiness, unhealthy, moneylosing, dreamblocking, pleasure work, sleep

4. 백엔드 컬럼 정의 (time_entries 테이블)

Time.js 필드 → DB 컬럼 매핑 · 사용자(user_id)별로 저장

컬럼명 한글명 타입 필수 Time.js 매핑
id ID UUID O 고유키 (자동생성)
user_id 사용자ID UUID O 로그인 사용자 (auth.users.id)
task_name 과제명 텍스트 O ← row.taskName
time_tracked 사용시간 string O ← row.timeTracked (hh:mm)
category 카테고리 string O ← row.category
productivity 생산성 string O ← row.productivity (productive/nonproductive/other)
entry_date 날짜 날짜 O ← row.date (YYYY-MM-DD)
feedback 피드백 텍스트 X ← row.feedback
created_at 등록일시 날짜시간 O 자동 생성
updated_at 수정일시 날짜시간 O 자동 수정

5. 시급 설정 (user_settings 또는 별도 테이블)

시간가계부 상단 "시급입력" 값은 사용자별로 저장. 가격 계산 시 사용.

컬럼명 한글명 타입 설명
user_id 사용자ID UUID 로그인 사용자
hourly_rate 시급 integer 원 단위 (예: 45000)

6. 저장 예시 (Time.js 입력 → DB 저장)

user_id task_name time_tracked category productivity entry_date feedback
user_abc123 건강한 식사 준비 00:30 health productive 2026-02-21
user_abc123 아이디어 작업 02:00 sideincome productive 2026-02-21 집중 잘됨
user_abc123 근무 08:00 work other 2026-02-21
→ 같은 user_id로 여러 행이 쌓입니다. entry_date로 날짜별 조회, category/productivity로 필터링 가능. 시급(hourly_rate)은 user_settings에 사용자별 1건 저장.

근무표 테이블 구조

WorkSchedule.js 기준 · 백엔드에 사용자별로 근무표 데이터가 저장되는 방식 · 불러오기/저장 시 충돌 최소화

1. WorkSchedule.js에서 사용하는 데이터 구조 (프론트엔드)

collectRowsFromDOM()이 반환하는 row 객체

필드명 한글명 타입 설명
name 근무시간 string 예: 08:00-17:00 (시간대 표기)
workType 근무유형 string 초과근무 | 조기퇴근 | 지급예정초과근무 | 온콜 | 온콜초과근무 등 (사용자 추가 가능)
hoursWorked Hours string 실제 근무 시간 수 (예: 2, -1, 0.5)
workDate 근무일 string YYYY-MM-DD
hours 시간적립 string 초과근무→+hoursWorked, 조기퇴근→-hoursWorked, 그 외→빈값 (자동계산)
memo 메모 string 자유 메모

2. 근무유형 (work_type) 규칙

근무유형 시간적립 계산 삭제 가능
초과근무 hours = hoursWorked × 1 X (보호됨)
조기퇴근 hours = hoursWorked × -1 X (보호됨)
그 외 (지급예정초과근무, 온콜 등) hours = 빈값 (총합에 미포함) O

3. 백엔드 테이블 정의 (work_schedule_entries)

사용자(user_id)별로 저장 · id로 행 식별하여 upsert 시 충돌 방지

컬럼명 한글명 타입 필수 WorkSchedule.js 매핑
id ID UUID O 고유키 (클라이언트 생성 또는 서버 자동생성)
user_id 사용자ID UUID O 로그인 사용자 (auth.users.id)
work_time 근무시간 텍스트 X ← row.name
work_type 근무유형 string O ← row.workType
hours_worked Hours numeric X ← row.hoursWorked
work_date 근무일 날짜 O ← row.workDate (YYYY-MM-DD)
hours_accrued 시간적립 numeric X ← row.hours (초과/조기만 저장, 그 외 null)
memo 메모 텍스트 X ← row.memo
sort_order 정렬순서 integer X 행 순서 유지 (0, 1, 2...)
created_at 등록일시 날짜시간 O 자동 생성
updated_at 수정일시 날짜시간 O 자동 수정

4. 근무유형 옵션 테이블 (work_schedule_type_options)

사용자별 커스텀 근무유형 저장 · 초과근무/조기퇴근은 기본 포함, 삭제 불가

컬럼명 한글명 타입 설명
user_id 사용자ID UUID 로그인 사용자
type_name 근무유형명 string 초과근무, 조기퇴근, 지급예정초과근무 등
sort_order 정렬순서 integer 드롭다운 표시 순서

※ 또는 JSON 배열로 user_settings에 저장: work_type_options: ["초과근무", "조기퇴근", "지급예정초과근무", ...]

5. 저장/불러오기 전략 (충돌 방지)

항목 권장 방식
행 식별 각 행에 UUID(id) 부여. 클라이언트에서 crypto.randomUUID()로 생성 후 저장
저장 upsert: id 있으면 UPDATE, 없으면 INSERT. user_id로 RLS 적용
불러오기 WHERE user_id = :current_user ORDER BY work_date, sort_order
동시 편집 행 단위 upsert이므로 다른 행 수정과 충돌 없음. 같은 행 동시 수정 시 updated_at 비교 후 최신 우선
삭제 DELETE WHERE id = :row_id AND user_id = :current_user

6. 저장 예시 (WorkSchedule.js 입력 → DB 저장)

id user_id work_time work_type hours_worked work_date hours_accrued memo
uuid-1 user_abc123 08:00-17:00 초과근무 2 2026-02-22 2
uuid-2 user_abc123 08:00-15:00 조기퇴근 1 2026-02-22 -1 병원
uuid-3 user_abc123 08:00-17:00 지급예정초과근무 2 2026-02-23 null
→ user_id로 사용자별 데이터 격리. id로 행 단위 upsert하여 불러오기/저장 시 충돌 최소화. work_date + sort_order로 정렬. hours_accrued는 초과근무/조기퇴근만 저장하며, 총 "적립된 총 시간"은 이 값들의 합으로 계산.
